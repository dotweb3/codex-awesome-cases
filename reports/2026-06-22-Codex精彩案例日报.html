Lupen：给 Codex/Claude Code 花费做“逐 turn 收据”

HNGitHubmacOS app
发布时间：2026-06-21（非当天新增，1 天前；v0.6.1 于 2026-06-22） · 来源：HN Show HN · GitHub · README 截图
88/ 100

为什么值得看

它不是又一个 token 总数面板，而是把 Codex 的本地 rollout JSONL 和 Claude Code 日志拆到 turn、step、skill group、sub-agent、附件来源，并重新计算费用差异。对高频使用 Codex 的团队，这类“账单可解释性”已经从好看面板变成工程治理需求。

Codex 具体做了什么

读取 $CODEX_HOME/sessions/**/rollout-*.jsonl，保留 reasoning/cached token 信息，支持用 codex resume 继续历史会话，并用 verifier 检查 Codex 记录的成本漂移。

最有脑洞 / 最实用的点

最实用的是“费用归因到 sub-agent/附件来源”：能看出到底是某个工具循环、图片、URL，还是长思考步骤吃掉了预算。

可借鉴做法

  • 给 Codex 日志建本地索引时，先保留原始事件，再单独做规范化层。
  • 把 verify/budget 做成 CLI exit code，可接进 CI 或 pre-commit。
  • 花费报告按 provider 隔离，避免 Claude/Codex 混在一个总数里。

不确定点

HN 讨论量很低，社区热度主要来自 Show HN 发现和项目自身迭代；真实团队规模使用还需要后续 issue/release 观察。

agent-rollback：给 Codex CLI 补上可恢复安全层

redditGitHubMCP
发布时间：2026-06-15 左右（非当天新增，约 7 天内） · 来源：reddit 发现帖 · GitHub · MCP.Pub 条目
84/ 100

为什么值得看

很多 Codex 事故不是“模型不会写”，而是它在提交前几分钟改了 6 个文件、用户没有干净回退点。这个项目把 checkpoint、diff、restore、MCP tool 和 Codex hooks 组合成一条低成本恢复链。

Codex 具体做了什么

通过 Codex hook 自动在风险操作前后做内容寻址快照；用户可以在 Codex 对话里说“go to last checkpoint”，由 MCP server 执行恢复、diff、pin、prune。

最有脑洞 / 最实用的点

它把“用 git 就好”的笼统建议拆成 agent 专用的时间线：pre-commit 之前也能 per-op undo，且不要求当前目录已经有干净 git 工作流。

可借鉴做法

  • 任何让 Codex 自动改文件的流程都应先定义恢复粒度。
  • MCP 自然语言入口比记命令更适合紧急回滚。
  • 恢复工具要默认本地、无遥测，避免把事故现场再上传。

不确定点

reddit 页面在本环境被网络策略拦截，正文使用搜索索引片段和 GitHub/MCP.Pub 交叉验证；README 中提到 Codex 内置 /rewind 的边界仍在演进。

Ray-Ban Display 上的 Snake：Codex 进入可穿戴 Web App 原型

XGitHubMeta toolkit
发布时间：2026-06-15 左右（非当天新增，约 7 天内） · 来源：X 文章（直连受限） · Meta Wearables toolkit · Snake 示例目录
83/ 100

为什么值得看

这是“用 Codex 写一个网页小游戏”之外更有信息量的案例：目标设备是 600x600 透明显示、D-pad/EMG 手势输入、无依赖 Web App。Codex 的价值在于把新硬件约束变成可运行原型。

Codex 具体做了什么

X 索引显示作者称用 Codex 为 Ray-Ban Display 眼镜构建 Snake；Meta 官方 incubator repo 同时提供 Claude Code、Codex、Cursor、Copilot 插件和 Snake 示例代码。

最有脑洞 / 最实用的点

把 AI coding skill 写成硬件设计约束包：viewport、黑底透明、高对比、focusable、D-pad 控制。模型不是自由发挥 UI，而是在设备规则内生成。

可借鉴做法

  • 给 Codex 的 skill/插件里明确硬件输入方式和显示限制。
  • 用浏览器先模拟 D-pad，再部署到真实设备。
  • 示例应用要小而完整，Snake 比“天气 demo”更能暴露输入、状态、碰撞、分数等细节。

不确定点

X 原文在无登录环境只能看到搜索索引，无法核验完整 thread；主证据以 GitHub 示例和可运行代码为准。

Tina Austin：把 Codex 当教育工作流引擎，而不是聊天框

Substackeducationworkflow
发布时间：2026-06-16（非当天新增，6 天前） · 来源：Substack 原文 · 作者 Substack · 评论页
82/ 100

为什么值得看

这不是开发者炫技，而是非软件工程场景的具体交付清单：整理 LMS 导出、自动周公告、修复网站后端、做可访问性审计、把演示稿重构成论文脚手架、生成 workshop 材料和 docx facilitator guides。

Codex 具体做了什么

作者把 Codex 描述为跨文件、文件夹、文档、应用和重复流程的工作空间，让它组织 LMS、生成资源、修复结构问题，并在后台跑教育行政任务。

最有脑洞 / 最实用的点

“Codex drafts, organizes, revises, and automates. You approve.” 这个边界很有借鉴性：教师保留判断，Codex 做重复材料处理和流程化交付。

可借鉴做法

  • 把课程/工作坊拆成文件夹级项目，而不是单 prompt。
  • 让 Codex 先整理输入资产，再生成可复用模板。
  • 对高准确度内容保留人工决策树，避免把教育判断外包。

不确定点

作者列出的是个人经验清单，未公开完整仓库或可下载成品；因此入选为“工作流案例”，不是可复现开源项目。

Hermes Agent：把 Codex app-server 变成可嵌入运行时

XGitHubDocs
发布时间：2026-06-20/21（非当天新增，2 天内） · 来源：Hermes Codex runtime 文档 · GitHub · Codex skill 文件 · X 线索页
80/ 100

为什么值得看

它把 Codex 从“终端里的 agent”提升为别的 agent shell 可以调用的 runtime：Hermes 负责消息网关、记忆、skills 和调度，真正的 shell、apply_patch、MCP、审批和沙箱交给 Codex app-server。

Codex 具体做了什么

Hermes 可以把 openai/* 与 openai-codex/* turns 交给 Codex CLI app-server；Codex 原生插件、MCP server 配置和 app-server tool events 被迁移/桥接到 Hermes 会话里。

最有脑洞 / 最实用的点

这说明 Codex 生态正在出现“运行时层”分工：一个产品不必重写代码执行、文件编辑、approval、plugin migration，可以把 Codex app-server 作为底层代理引擎。

可借鉴做法

  • 要集成 Codex，优先研究 app-server 而不是屏幕抓取 TUI。
  • 把外层工具的 MCP 配置翻译成 Codex config.toml，并保留用户自定义块。
  • 长任务消息网关要单独处理进度事件，否则用户只看到沉默到最终结果。

不确定点

X 直连不可读，只能通过搜索索引确认近期讨论；文档是主证据。该功能仍标注 optional/beta，部分能力有明确限制。
