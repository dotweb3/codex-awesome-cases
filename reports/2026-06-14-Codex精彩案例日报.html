Circus Chief：从手机调度 Claude Code、Codex 和 Gemini 的代理控制台

HNGitHub非当天新增：2026-06-02

HN 作者把 Circus Chief 描述为面向小屏幕的 coding agent session 控制面板；帖子披露它支持 OpenAI Codex agents，并且作者已经 dogfood 到 2,184 次 agent session，其中 363 次是 Codex session。

92入选评分

为什么值得看

它把多个代理会话、PR、计划、命令和工作树隔离放进同一个移动友好控制台，适合观察“一个人同时管理多条 agent 工作流”的形态。

Codex 具体做了什么

Codex 是一等 agent 类型；HN 帖子称项目构建过程中加入 Codex，并统计出 363 个 Codex sessions。

最有脑洞/实用点

代理不仅被管理，还能操作 Circus Chief 自己：生成后续 session、安排任务、操作看板和响应结果。

可借鉴做法

给每次 agent run 独立 worktree、可链式模板、可重试调度和共享 canvas，让长时程任务有“操作台”而不是散落聊天。

证据点

  • Show HN 帖子：列出 session 数、PR 数和 Codex session 统计。
  • GitHub README：包含功能截图、安装命令和 Codex/Gemini/Claude 支持说明。

不确定点

HN 评论数不高；项目是否适合团队生产使用仍要看鉴权、日志隐私和长期维护。

AgentGraphed：把本地 Codex CLI 会话做成历史、搜索和成本仪表盘

HNGitHub非当天新增：2026-06-09

Show HN 作者痛点很具体：Claude/Codex 很强，但恢复旧 session、找历史上下文和向别人展示使用量都很麻烦。AgentGraphed 读取本地日志，生成可搜索的本地 SQLite 仪表盘。

88入选评分

为什么值得看

它不是又一个聊天壳，而是把 agent 的“工作记忆”变成可检索、可恢复、可分享的数据层。

Codex 具体做了什么

README 明确支持 Codex CLI，并读取 ~/.codex/sessions/YYYY/MM/DD/*.jsonl 作为数据源。

最有脑洞/实用点

从任意旧 session 一键生成新聊天 primer，把“我记得之前问过”变成可复用上下文。

可借鉴做法

把 agent 日志当产品数据库处理：索引、分类、成本估算、时间线、项目聚合，本地优先，AI 摘要可选。

证据点

  • Show HN 帖子：作者列出恢复 session、搜索历史、分享统计等问题与解决方式。
  • GitHub README 与 产品页：说明本地 SQLite、Codex 日志目录和截图。

不确定点

自动分类和生成上下文需要自带 LLM key；隐私边界取决于用户是否开启这些可选功能。

Slipstream：给 Codex/Claude Code/Cursor 做上下文压缩与 token 仪表盘

RedditProduct新近：约 2 天前

Reddit r/SideProject 新帖把 Slipstream 定位成“一键安装、看得见节省效果”的上下文压缩桌面应用。产品页明确写支持 Claude Code、Codex、Cursor、Aider 和 Copilot，并声称 60–95% token 降幅。

83入选评分

为什么值得看

它击中代理工具的真实运营问题：长上下文、高 token、不可见的成本。截图里把 processed、saved、cost reduction 直接做成控制面板。

Codex 具体做了什么

Codex 是目标 agent 之一；Slipstream 在模型调用前压缩 agent context，属于 Codex 使用链路上的成本/上下文治理工具。

最有脑洞/实用点

用桌面代理层接入多种 agent，而不是要求每个项目单独手工配置 token 节省策略。

可借鉴做法

把“省 token”做成可审计的产品体验：按 agent、模型、日期展示节省量和成本估算，降低团队采用门槛。

证据点

  • Reddit r/SideProject 帖：搜索索引显示发布约 2 天前，标题直接提到 Claude Code 和 Codex。
  • 产品页：meta description 与页面文案写明支持 Codex，并展示 token/cost 仪表盘。
  • Headroom GitHub：产品页标注 built on headroom，作为底层思路补充。

不确定点

Reddit 原帖 JSON 直连被网络安全页阻断，热度只能通过搜索索引和可见摘要判断；token 降幅尚未独立复测。

Lark Bridge：不用 ChatGPT mobile，直接从 Telegram/iMessage 控制 Codex CLI

HNGitHub当天发现：2026-06-13 UTC

OpenAI 已有移动控制路径，但 Lark 的切入点更朴素：让手机里已有的聊天应用变成 Codex CLI 入口。HN 作者称它连接 Codex CLI 到 Telegram 或 iMessage，用于暂停、停止、重试、查状态等控制。

80入选评分

为什么值得看

它把 Codex CLI 降到“发消息给 bot”这种日常入口，适合碎片时间远程推进本地 agent 任务。

Codex 具体做了什么

README 写明当前 build 以 OpenAI Codex CLI 作为 active AI backend，消息被转发给配置好的 Codex CLI，再把结果发回同一聊天。

最有脑洞/实用点

iMessage 支持让 macOS 用户可以不安装新 mobile app，只用 Messages.app 的 AppleScript 轮询。

可借鉴做法

把控制命令设计成聊天命令：/pause、/resume、/stop、/retry、/clear，降低远程控制心智负担。

证据点

  • Show HN 帖子：2026-06-13 发布，说明不依赖 ChatGPT mobile app 的动机。
  • GitHub README：列出 Codex CLI、Telegram、iMessage 支持和命令。

不确定点

HN 评论数为 0，项目仍早期；iMessage 自动化可靠性取决于 macOS 权限和 Messages.app 状态。

黑洞模拟：Codex 参与生成、测试、改进天体物理数值算法

OpenAIYouTube新近：2026-06-11

本期最“越界”的案例来自 OpenAI 与 University of Arizona 的交叉来源：Chi-kwan “CK” Chan 用 Codex 辅助生成和测试模拟黑洞周围电子、离子运动的数值方法，把原本缓慢的算法探索压缩到更短周期。

90入选评分

为什么值得看

它不是常见 CRUD 或脚手架，而是把 Codex 放进科研算法探索：产生候选方案、测试、筛选，再由研究者判断。

Codex 具体做了什么

OpenAI 索引摘要写明 Codex 帮助 Chan refine and test algorithms；UArizona 也写到用 AI-assisted workflows 生成、测试、改进数值方法。

最有脑洞/实用点

“写代码”变成“探索算法空间”：Codex 不只是实现已知方案，而是给研究者提供可检验候选。

可借鉴做法

把 agent 输出限定为可运行、可比较、可复现实验；让领域专家审算法含义，Codex 负责高通量实现与测试。

证据点

  • OpenAI 案例页 与 官方视频：描述 Codex 用于黑洞模拟算法探索。
  • University of Arizona 新闻：独立复述 AI/Codex 辅助生成、测试、改进数值方法。

不确定点

OpenAI 原文页面在本地无头访问触发 Cloudflare，正文信息来自搜索索引与可访问的二级来源交叉验证。

今日灵感清单

  • 给 Codex session 建一个“控制面板”，而不是只依赖终端滚屏：状态、PR、工作树、后续任务都应结构化。
  • 把 ~/.codex/sessions/ 当作产品数据源：搜索、复盘、恢复上下文、成本估算都能从这里长出来。
  • 移动端 Codex 不一定要新 app，Telegram/iMessage 这种已存在入口也能成为轻量控制层。
  • 对长上下文代理，成本优化必须可见：只说“压缩”不够，应该能展示每个 agent/model 的节省证据。
  • 在科研或复杂算法场景里，让 Codex 产出“可检验候选”，比让它直接给最终答案更稳。
